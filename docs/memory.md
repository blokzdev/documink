# DocuMink — Mink Memory Architecture

**Scope:** authoritative specification for Mink's memory subsystem
**Related:** Blueprint §3.2, §5.6; PRD §5.4; Roadmap V1 Phase 12, V1.2 Phases 4–6

This document defines how Mink remembers. It is the source of truth if anything in the PRD, Blueprint, or Roadmap conflicts with statements here about memory behavior.

---

## 1. Design goals and non-goals

### 1.1 Goals

- **Persistent personalization.** Mink remembers user preferences, workflows, and context across sessions and across the user's paired devices.
- **Consistent experience across devices.** The same Mink on phone, laptop, and (post-V4) shared documents reasons from the same memory state.
- **Privacy-preserving memory.** Memory rows never contain raw PII/PHI plaintext. The vault's security guarantees extend to the memory layer.
- **Tier-scaled sophistication.** Memory quality adapts to the active Tier 4 model — high tiers get autonomous multi-type memory updates, low tiers get explicit-only memory.
- **User control and transparency.** Every memory is inspectable, editable, deletable, and provenance-tagged. Users can forget individual facts, categories, or everything.
- **Project-scoped isolation.** Memory in one Project is invisible to another Project unless the user explicitly performs a cross-Project action.
- **On-device only.** Memory never leaves the user's paired devices. Sync via CRDT on the existing sync channel; no cloud inference or cloud storage of memory content.

### 1.2 Non-goals

- **Not a database for user documents.** The `documents` table is the document store; memory records *about* documents, not their contents.
- **Not an arbitrary-fact store.** Memory is structured into six specific types; arbitrary JSON blobs go elsewhere.
- **Not a multi-agent system.** Single Mink per user; memory operations route through one LLM with deterministic tool dispatch. See ADR-012.
- **Not a shadow tokenization system.** Memory only tracks entities that are already tokenized in the vault. It never creates implicit token records for un-redacted PII.

## 2. Memory taxonomy

Adapted from MIRIX (Wang & Chen, 2025) with architectural changes for on-device operation. Six types; three active in V1, three schema-ready with activation in V1.2.

### 2.1 Core Memory (active V1)

**What it is:** stable identity facts and preferences about the user that rarely change.

**Examples:**
- "User prefers concise responses."
- "User is a privacy attorney."
- "User's default tone is formal."
- "User dislikes emojis in responses."

**Schema:** `mink_core_memory` (see Blueprint §3.2).

**Retrieval:** direct key lookup. Always loaded into Mink's context when a chat session starts.

**Update triggers:**
- Explicit user statement parsed by Mink ("I prefer X")
- User sets preference directly in Settings
- System defaults from Project templates (e.g., Medical Records template sets `user_preferred_tone=conservative_clinical` initially)

**Scope:** both global (`project_id=NULL`) and Project-scoped. Project-scoped core memory lets the user have different personas per Project ("formal in Tax Documents, creative in Novel Draft").

### 2.2 Episodic Memory (active V1)

**What it is:** time-stamped summaries of what happened, organized by Project.

**Examples:**
- "On 2026-04-21 you scanned 12 documents in Medical Records and tokenized 47 MRNs."
- "On 2026-04-22 you exported a redacted PDF of the Q1 tax return."
- "Last Tuesday you asked Mink to find all documents mentioning `<PERSON_a7x9qB>` — found 14."

**Schema:** `mink_episodic_memory`.

**Retrieval:** time-range query plus optional topic filter. For example, `recall_episodic(since: -7d, project_id: currentProject)` returns the last week of activity in this Project.

**Update triggers:**
- Automatic after user actions: scan completed, detection run, redaction applied, export generated, chat session closed with significant activity.
- Summary generated by Mink from the action context (tier-scaled — see §7).
- User can explicitly ask Mink to "remember that we did X today."

**Scope:** global or Project-scoped. Global episodes capture cross-Project activity like "on 2026-04-23 you created the Creative Writing Project."

### 2.3 Knowledge Vault (active V1, already implemented)

**What it is:** verbatim PII/PHI that must be preserved exactly — the vault's existing `tokens` table.

**Schema:** existing `tokens` table from Blueprint §3.1. No new structure needed.

**Retrieval:** HMAC-SHA256 fingerprint lookup for Mink's internal reasoning (no plaintext materialized). Full decode via `decode_token` tool with biometric gate.

**Update triggers:** existing tokenization flow — whenever a user applies a reversible operator (Token-Random, FPE, Encrypt), a row is written to `tokens`. No new write path from memory layer.

**Scope:** per-document (hence per-Project via document's `project_id`).

**Critical property:** this is the *only* place plaintext PII lives. All other memory types reference it via fingerprint, never by value.

### 2.4 Semantic Memory (schema V1, activation V1.2)

**What it is:** entities and their relationships. Who is X, where is Y, how is X related to Y. Stored hierarchically with typed relationships.

**Examples:**
- "Dr. Chen" (PERSON) — canonical fingerprint matches `tokens` row; descriptor "cardiologist"; parent "Medical Providers"; relationships: works_at → Stanford Medical, mentioned_in → 14 documents.
- "Stanford Medical" (ORGANIZATION) — descriptor "hospital in Palo Alto"; children: Dr. Chen, Dr. Kim, Dr. Lee.
- "Quarterly Review" (CONCEPT) — no canonical fingerprint (not PII); referenced in several Engineering docs.

**Schema:** `mink_semantic_memory` + `mink_semantic_relationships`.

**Retrieval:** three patterns:
1. **Fingerprint lookup** — "I just saw 'Sarah Chen' in a document; do I already know this person?" → HMAC match against `canonical_fingerprint`.
2. **Descriptor match** — "Find entities described as 'cardiologist'" → FTS5 on descriptor.
3. **Vector similarity** — "Find entities related to 'oncology'" → sqlite-vec ANN on embedding.
4. **Relationship traversal** — "Who works at Stanford Medical?" → recursive CTE on `mink_semantic_relationships`.

**Update triggers:**
- Document processing: when a token is written to the vault, check if an entity with matching fingerprint exists. If yes, increment `occurrence_count` and update `last_seen_at`. If no, create new entity row with descriptor inferred from context.
- Chat: when Mink discusses an entity and learns new facts ("Dr. Chen moved to Mayo Clinic"), update descriptor or add new relationships.
- User-initiated: "Remember that Dr. Chen is my referring physician."

**Scope:** primarily Project-scoped. Global semantic memory exists but is rare and always user-initiated (e.g., "Remember globally that Dr. Chen is my referring physician across all my medical work").

**PII safety:** every entity with `canonical_fingerprint != NULL` has its plaintext in the `tokens` vault. Memory row stores only fingerprint + descriptor + relationships — never plaintext.

### 2.5 Procedural Memory (schema V1, activation V1.2)

**What it is:** observed patterns in how the user works — trigger conditions and typical responses.

**Examples:**
- "When user scans a tax return document, they typically apply FPE to EIN and Token-Random to PERSON."
- "When user adds a document to Medical Records Project, they typically also run detection immediately."
- "User usually exports redacted PDFs at end of Project session."

**Schema:** `mink_procedural_memory`.

**Retrieval:** pattern match on current action context. `recall_procedural(action: 'scan', document_type: 'tax_return')` returns observed patterns for this situation.

**Update triggers:**
- Pattern observation: after the same trigger→response sequence is observed N times (configurable, default 3), increment `observed_count` and raise `confidence`.
- User confirmation: when a pattern reaches confidence threshold, Mink surfaces it for user confirmation ("I've noticed you FPE EINs on tax returns — want me to do this automatically?"). User confirms → `user_confirmed=1`, automation active. User rejects → pattern marked deleted.

**Scope:** Project-scoped (patterns learned in one Project don't apply to another by default). Global patterns exist for cross-Project habits ("user always reviews audit log on first unlock of day").

**PII safety:** procedural memory stores patterns (entity types, operators, action contexts) — never specific PII values. "User FPEs EINs" is a pattern; "User FPEs EIN 12-3456789" would be a PII leak into memory (forbidden).

### 2.6 Resource Memory (schema V1, activation V1.2)

**What it is:** documents and files shared by the user, with semantic retrieval.

**Schema:** existing `documents` table + embedding entries in `mink_embeddings` for document summaries.

**Retrieval:** hybrid BM25 (FTS5) + vector similarity on document summaries.

**Update triggers:**
- Document added to vault → compute summary embedding → store in `mink_embeddings`.
- Document modified → recompute embedding.
- Backfill on V1.2 activation for documents created pre-activation.

**Scope:** inherits document's `project_id` — Project-scoped.

**PII safety:** embeddings computed on *redacted* document text when available, else on document metadata (title, type, size, creation date). Never on unredacted plaintext.

## 3. PII-safe reference model (critical invariant)

The memory layer's central security property is that **raw PII/PHI plaintext never enters memory tables**. This section defines the mechanism that enforces it.

### 3.1 The reference model

Every reference to a PII/PHI entity in memory uses one of three forms:

**Form A — Token reference in structured content:**
```json
{
  "type": "token_ref",
  "token_id": "tok_01HXJ4K2Z6F3X9M2N5Q7V8W1E3",
  "display_fallback_type": "PERSON"
}
```
Stored inline in JSON payloads (e.g., `mink_core_memory.value_json`, `mink_episodic_memory.details_json`). The `display_fallback_type` lets the UI render `<PERSON>` when masked and a biometric-gated decode is needed for reveal.

**Form B — Inline token marker in text:**
```
"I found 14 mentions of <<tok_01HXJ4K2Z6F3X9M2N5Q7V8W1E3>> across your documents."
```
Stored in `chat_messages.content` and `mink_episodic_memory.summary`. The `<<tok_...>>` marker is a reserved syntax; the rendering layer replaces it with the appropriate display value based on mask state.

**Form C — Canonical fingerprint (Semantic Memory only):**
```sql
canonical_fingerprint = HMAC-SHA256(plaintext, fingerprint_hmac_key)
```
Stored directly in `mink_semantic_memory.canonical_fingerprint` as a 32-byte BLOB. Deterministic for a given plaintext but reveals nothing about it. Used for fast lookup ("do I already have an entity for this?") without needing the plaintext.

### 3.2 Fingerprint-based entity identity

When processing a document, a new token is created in `tokens`. Its `plaintext_fingerprint` (already computed per Blueprint §8.1) is the entity's stable identity. The Semantic Memory update flow:

```
On new token creation for plaintext P:
  fingerprint ← HMAC(P, fingerprint_hmac_key)   // already computed by vault
  entity ← SELECT * FROM mink_semantic_memory
           WHERE workspace_id = ? AND canonical_fingerprint = fingerprint
  IF entity EXISTS:
    UPDATE entity SET occurrence_count += 1, last_seen_at = now()
  ELSE:
    descriptor ← infer_descriptor_from_context(surrounding_document_text)
    INSERT INTO mink_semantic_memory (..., canonical_fingerprint=fingerprint, descriptor, ...)
```

Note that `infer_descriptor_from_context` runs the Tier 4 LLM on surrounding **redacted** text — the full plaintext is never needed to infer "this appears to be a cardiologist." The LLM sees "Dr. [PERSON] is a cardiologist at [ORGANIZATION]" and outputs `descriptor: "cardiologist"`.

### 3.3 Write-path enforcement

All memory writes go through `MemoryRepository` which enforces the no-plaintext invariant:

```dart
class MemoryRepository {
  Future<void> write(MemoryType type, dynamic content) async {
    final plaintextCheck = await _scanForUnreferencedPii(content);
    if (plaintextCheck.hasViolations) {
      throw MemoryPiiLeakError(
        message: 'Memory write rejected: content contains unreferenced PII. '
                 'Convert to token references first.',
        violations: plaintextCheck.violations,
      );
    }
    // proceed with write
  }
}
```

`_scanForUnreferencedPii` runs the detection pipeline on incoming content. Any detected PII that is not already a token reference is a violation. Integration tests assert this by attempting to write plaintext names, addresses, MRNs, SSNs directly into each memory table — all must fail with structured errors.

### 3.4 The three runtime paths for PII in memory

**Path 1 — Mink reasons about an entity without materializing plaintext.** Most common case.

Example: User asks "how many times have I referenced Dr. Chen?"
- Mink identifies "Dr. Chen" in the query, invokes `recall_semantic(entity_type: 'PERSON', descriptor: 'Chen')`.
- Semantic Memory returns entity with `occurrence_count: 14` and list of document IDs.
- Mink responds: "I've found 14 mentions of `<<tok_01HXJ4...>>` across your documents."
- UI renders the token as `<PERSON_a7x9qB>` in default masked state, or "Dr. Sarah Chen" when user reveals.
- Plaintext never enters Mink's context window.

**Path 2 — User explicitly invokes decode.** Rare, always user-initiated.

Example: User asks "what's Dr. Chen's office number?"
- Mink's query references a token.
- Mink invokes `decode_token(token_id, reason: 'user_requested')`.
- Biometric gate triggered; user authenticates.
- Plaintext loaded into transient response buffer (never persisted to memory, never written to `chat_messages.content`).
- Response rendered ephemerally; when user navigates away, plaintext gone from memory.
- `chat_messages.content` stores the token reference, not the decoded value.
- Audit log entry written.

**Path 3 — Fingerprint-based search without materialization.** Internal tool use.

Example: User asks "are there other documents mentioning this person?"
- Mink extracts fingerprint from the referenced token.
- `search_by_fingerprint(fingerprint)` returns list of document IDs containing matching tokens.
- Plaintext never touched.

### 3.5 What happens if a user tells Mink a PII value in chat

Edge case: user types "Dr. Chen's number is 555-1234" directly into chat.

Handling:
1. On message submission, chat input runs through detection pipeline.
2. Detection flags "555-1234" as PHONE.
3. UI prompts user: "You've entered what looks like a phone number. Should I (a) store this and tokenize for future reference, (b) redact before sending to Mink, or (c) send as-is for this one response?"
4. Option (a) creates a vault token, replaces the plaintext with a token reference in the chat message, then sends to Mink.
5. Option (b) redacts the value before Mink sees it.
6. Option (c) sends the plaintext to Mink's context for this turn only, but `chat_messages.content` still stores the sanitized version (replaced with `[REDACTED_BY_USER]` marker).

This keeps the invariant intact: even user-provided PII doesn't pollute memory tables unless explicitly tokenized first.

## 4. Router architecture (single-agent, deterministic)

MIRIX proposes a multi-agent architecture with eight LLMs (six memory managers + meta + chat). For on-device operation on tiered hardware, this is untenable — eight cold-start LLM invocations per response would destroy latency and battery life on anything below flagship tier. DocuMink uses a **single-agent deterministic router** that achieves the taxonomic benefits without the multi-agent overhead.

### 4.1 Components

- **Mink (single LLM instance)** — the active Tier 4 model. Sees user input, decides what to respond with, emits tool calls.
- **Router (deterministic code, not LLM)** — dispatches memory operations based on tool call names and parameters. No LLM involvement in routing.
- **Memory tools (exposed to Mink via ToolRegistry)** — named tools like `recall_core`, `recall_episodic`, `recall_semantic`, `recall_procedural`, `remember`, `forget`.
- **Memory repositories (per type)** — database access layer per memory type; each implements read, write, update, delete with PII-safety enforcement.

### 4.2 Query flow

```
User message → Mink (LLM)
  ├─ If Mink wants to remember something: emits `remember(type, value, scope)` tool call
  │    → Router dispatches to MemoryRepository.write()
  │    → PII-safety check
  │    → Typed store insert
  │
  ├─ If Mink wants to recall: emits `recall_<type>(params)` tool call
  │    → Router dispatches to corresponding repository
  │    → Results returned to Mink as tool_result
  │    → Mink incorporates into next response
  │
  └─ If Mink wants to reference an entity it knows about: includes token ref
       in response text, which the UI renders based on mask state.
```

Router is ~200 lines of Dart. No LLM calls. Fast, deterministic, debuggable.

### 4.3 Context assembly for chat

When a chat session turn begins, `ContextAssembler` builds the LLM input:

1. **System prompt** — from Project manifest if in Project, else default.
2. **Core Memory slice** — all Core Memory entries scoped to current Project + globals. Always loaded (small — a few kB typically).
3. **Recent Episodic summaries** — last 7 days in current scope, capped at ~10 entries.
4. **Relevant Semantic entities** — entities referenced in the last 3 turns (pre-fetched).
5. **Active Procedural patterns** — user-confirmed patterns matching current context.
6. **Chat history** — last N turns windowed to fit model context budget.
7. **Tool descriptions** — filtered by Project permission manifest.
8. **Optional document snippets** — only if Project grants `read_documents` and user query references documents; vector-retrieved via Resource Memory.

Total context budget is tier-scaled per §7. On Minimum tier, ContextAssembler skips Semantic, Procedural, and document snippets entirely.

## 5. Retrieval patterns per type

| Memory type | Primary retrieval | Secondary retrieval | Index |
|---|---|---|---|
| Core | Direct key lookup | All-entries scan (small) | PK + `(workspace_id, project_id, key)` unique |
| Episodic | Time-range query | FTS5 on summary + topic filter | `(workspace_id, project_id, occurred_at)` |
| Knowledge Vault | HMAC fingerprint lookup | Token ID direct | `(workspace_id, plaintext_fingerprint)` |
| Semantic | Fingerprint match | Descriptor FTS5, vector similarity, relationship CTE | `(workspace_id, canonical_fingerprint)` + FTS5 index + vec index |
| Procedural | Pattern JSON match | Context-similarity | JSON extraction index |
| Resource | Vector + BM25 hybrid | Metadata filter | FTS5 on document text + vec index on summary |

## 6. Project scope enforcement

Every memory table has `workspace_id` and `project_id` columns. Isolation is enforced at the repository layer:

```dart
class EpisodicMemoryRepository {
  Future<List<Episode>> recall({
    required String workspaceId,
    String? projectId,           // NULL for global-only
    DateTime? since,
    String? topicFilter,
  }) async {
    // query always includes: WHERE workspace_id = ? AND
    //   (project_id = ? OR (project_id IS NULL AND :includeGlobal))
    // NO method accepts "all projects" without AuditedCrossProjectAccess.
  }
}
```

**Cross-project access is a separate audited API:**

```dart
class AuditedCrossProjectAccess {
  Future<List<T>> queryAllProjects<T>({
    required String reason,              // user-provided or tool-provided
    required Future<List<T>> Function(String projectId) perProjectQuery,
  }) async {
    await auditLog.write(event: 'cross_project_access', reason: reason);
    // execute perProjectQuery across all accessible projects
  }
}
```

Integration tests assert: memory written in Project A is unreadable from a session scoped to Project B unless `AuditedCrossProjectAccess` is invoked.

## 7. Tier-scaled update behavior

Memory quality adapts to the active Tier 4 model. The pattern: **higher tiers do autonomous, structured memory updates; lower tiers do explicit, minimal memory updates.**

| Tier | Core updates | Episodic updates | Semantic updates | Procedural updates |
|---|---|---|---|---|
| System-provided / Performance | Full — Mink infers preferences from conversation | Full — rich summaries with entity references | Full — auto-discovery + relationship graph | Full — pattern observation + confidence tracking |
| Standard | Full | Full | Full — but fewer speculative relationships | Full |
| Light | Inferred from explicit statements only | Compact summaries | Fingerprint match + basic descriptor only | Count observations; surface for user confirmation |
| Ultra-light | Explicit user statements only | Minimal — action type + timestamp | Fingerprint match only (no new descriptors) | Disabled |
| Minimum | Only when user says "remember that ..." | Disabled | Disabled | Disabled |
| Below Minimum | Disabled (Mink UI replaced) | Disabled | Disabled | Disabled |

Users on low-tier devices still get meaningful memory — they just don't get speculative auto-inference. When they upgrade devices, paired sync brings richer memory to the lower-tier device from the higher-tier device, and inference runs on the capable device.

## 8. User controls and transparency

### 8.1 Settings → Mink Memory UI

Six collapsible sections (three active V1, three activated V1.2):

- **Core Memory** — list of preferences and identity facts; inline edit.
- **Episodic Memory** — chronological activity log; filterable by Project and time; delete individual episodes or entire ranges.
- **Knowledge Vault** — already exposed via existing vault browser; linked from here.
- **Semantic Memory** — tree view of entities; tap entity to see relationships and source documents (token ref rendering); edit descriptors; delete entities (soft delete with undo).
- **Procedural Memory** — list of observed patterns with confidence; toggle user-confirmed status; delete patterns.
- **Resource Memory** — list of documents with similarity search; delete removes document from memory but (optionally) retains it in vault.

### 8.2 Per-entry provenance

Every memory entry shows one of:

- **"You told me"** — explicit user statement parsed by Mink.
- **"You set this"** — user configured directly via Settings.
- **"Inferred from conversation"** — Mink derived this from chat (confidence shown).
- **"Observed from your actions"** — Mink observed a pattern (confidence + count shown).
- **"System default"** — from a Project template.

### 8.3 Global actions

- **Export all memory** — structured JSON download covering all six types.
- **Forget everything in [Project]** — deletes all memory entries scoped to a Project; cascades to token references where appropriate.
- **Forget everything globally** — nuclear option; confirms twice; deletes all memory across all Projects.
- **Forget topic** — Mink-assisted flow: user describes what to forget in natural language; Mink surfaces matching entries for user confirmation before deletion.

### 8.4 Show/hide-masked rendering

This is a user control that applies uniformly across documents, chat, and memory views. Three default behaviors the user can configure:

- **Always masked** (default) — tokens render as `<PERSON_a7x9qB>` or user's preferred token display style. Reveal requires explicit tap + biometric if session not already authenticated.
- **Reveal on authenticated session** — when user unlocks the vault, tokens auto-reveal in all views until session lock. Tap to mask individually.
- **Ask per view** — when opening any view with tokens, prompt: "Show decoded values?"

The per-view override is session-scoped: reveal in Document A doesn't auto-reveal in Document B.

On copy-to-clipboard: always prompt "Copy masked or decoded?" with masked as default, regardless of current reveal state.

### 8.5 Transparency in chat

When Mink performs memory operations, they're visible inline in the chat transcript:

```
You: Do I have any other documents about Dr. Chen?

Mink: [🔍 recalled 1 semantic entity, searching documents...]
I found 12 other documents mentioning Dr. Chen across your Medical
Records project. Would you like me to list them?

[Mink learned: you confirmed Dr. Chen as a frequent entity]
```

Memory operations that change state (remember, forget, update) are audit-logged with event type `mink_memory_write`, `mink_memory_update`, `mink_memory_delete`.

## 9. Sync behavior per type

All six types sync via the existing CRDT infrastructure (cr-sqlite). But each type has different conflict semantics:

| Type | Conflict resolution | Rationale |
|---|---|---|
| Core Memory | Last-write-wins per key | Preferences should reflect latest user state |
| Episodic Memory | Set union (append-only) | Events happen independently on different devices; no conflicts by construction |
| Knowledge Vault | Set union of tokens; fingerprint dedup | Same fingerprint from different devices is the same entity |
| Semantic Memory | CRDT merge: entity by fingerprint; `occurrence_count` = max across replicas; `last_seen_at` = max; descriptors = LWW | Entity identity via fingerprint is deterministic; counts are monotonic; descriptors are user-editable |
| Procedural Memory | `observed_count` = sum across replicas; `user_confirmed` = OR; deletions = tombstone | Pattern observation happens independently; user confirmation on any device applies everywhere |
| Resource Memory (embeddings) | Embeddings deterministic from content + embedding model version; compute locally per device | No cross-device merge needed if both devices ran same model; version mismatch triggers recompute |

Sync state itself is encrypted with the same sync transport key as the rest of the vault. Memory never travels in plaintext between devices.

## 10. Threat model (memory-specific)

### 10.1 Defended against

- **Memory database exfiltration.** If Mink's memory tables were somehow copied without the SQLCipher key, attacker sees: HMAC fingerprints, non-PII descriptors, occurrence counts, relationships, timestamps, embedding vectors. No plaintext PII recoverable without the vault encryption key.
- **Forensic recovery of deleted memories.** SQLCipher uses overwriting deletion; tombstones + VACUUM remove residue. Deleted memory entries don't linger in recoverable form.
- **Cross-Project memory leakage via bugs.** Repository-layer filtering is the primary defense; integration tests assert Project A data invisible from Project B scope.
- **Memory poisoning via malicious Project template.** Project manifests are signed (Blueprint §6.4); unsigned or invalid-signature templates are rejected before creating a Project.

### 10.2 Out of scope

- **Statistical inference from memory metadata.** An attacker with access to the memory database (but not the vault) could infer patterns — "user has 47 entities of type PERSON in Medical Records project, suggesting significant medical history." This is a metadata leak we don't attempt to defend against; we note it in the privacy policy.
- **Nation-state with chip-level access.** Out of scope across the product.
- **User explicitly choosing to share memory.** Export-all-memory produces structured JSON; if a user shares it voluntarily with a third party, that's the user's choice.

### 10.3 Specific mitigations

- **Canonical fingerprint uses a separate HMAC key** (not the vault decode key) so that a compromise of the fingerprint key doesn't decrypt vault contents, and vice versa.
- **Embeddings are computed on redacted text** where possible; only non-PII portions of context feed the embedding model.
- **Memory audit events** are categorically separated from document audit events; a memory-focused export or review doesn't accidentally surface document decode history.

## 11. Open questions (for V1.2 activation and beyond)

These are intentionally left open for V1.2 design sessions, informed by real V1 usage data:

1. **How aggressive should automatic Semantic Memory creation be?** Conservative (only from vault tokens) vs. more aggressive (create entities from repeated chat mentions). V1 ships conservative; V1.2 considers opt-in more aggressive mode.
2. **How does Procedural Memory handle contradictory patterns?** If user FPEs EINs sometimes and Token-Random other times, what's the right UX for surfacing the ambiguity?
3. **What happens when a user deletes a vault token referenced by Semantic Memory?** Cascade-delete the entity? Keep the entity with `canonical_fingerprint = NULL`? Current plan: cascade-delete with user confirmation, but this needs UX validation.
4. **Should semantic relationships sync across Projects?** If "Dr. Chen works at Stanford" is learned in Medical Records, should it be available in a (different user) Research Project that also references Dr. Chen? Current plan: no — Project isolation wins — but this needs user feedback.
5. **How does memory interact with V4 URL-shared documents?** When a user receives a shared document, should Mink's memory ingest it automatically into the current Project? Current plan: yes, with explicit consent at import time.

## 12. "Don't do" reference (memory-specific)

1. **Don't write PII/PHI plaintext to any memory table.** Use token references or canonical fingerprints. `MemoryRepository.write` rejects plaintext PII; don't bypass it.
2. **Don't create memory entries for un-tokenized PII.** Only track entities that have been redacted via the user's own action. No shadow profiles.
3. **Don't route memory operations through the LLM.** Router is deterministic code. LLM emits tool calls; router executes them.
4. **Don't introduce new memory types without updating this doc.** Six types is the architecture; adding a seventh requires ADR-level discussion.
5. **Don't let one memory type's query leak into another's scope.** Each repository is bounded to its table; cross-type queries go through the router.
6. **Don't export memory without explicit user action.** No automatic exports, no telemetry, no crash-report inclusion of memory content.
7. **Don't treat Mink's inferred memory as ground truth.** Inferred provenance always shows confidence; user can edit or reject.
8. **Don't skip the show/hide-masked rendering layer.** Any UI surface displaying token references must go through the standard renderer so mask state is consistent.
9. **Don't hard-code tier-scaled behavior in LLM prompts.** Tier scaling is structural — which context is assembled, which tools are exposed, which memory updates are triggered. The LLM shouldn't be asked "are you on a low tier?" in the prompt.
10. **Don't merge memory across users.** When Teams tier ships (V3), shared vaults have a separate memory architecture decision; individual Mink memories do not auto-merge.
