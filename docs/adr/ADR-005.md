# ADR-005: Argon2id + Keystore/DPAPI for key management

- Status: Accepted
- Date: 2026-06-13
- Context: Recorded in V0 Phase 3; formalizes a decision already in the specs.

## Context

The vault (ADR-003) is only as strong as its key management. We need to derive encryption keys
from a user passphrase resistant to GPU/ASIC brute force, keep the master key out of persistent
storage, and wrap subordinate keys with hardware-backed platform facilities on both Android and
Windows — without ever storing the passphrase or plaintext keys.

## Decision

Derive a **Master Key (MK) from the user passphrase via Argon2id** (64 MB, t=3, p=4, salt in
`vault_meta`). MK lives in RAM only and is zeroed on lock. Subkeys derive via HKDF-SHA256 with
domain separation: a KEK (wraps DEKs), a fingerprint-HMAC key (token indexing), and a sync
transport key. The KEK is wrapped by the platform store via `flutter_secure_storage`:
**Android Keystore (StrongBox where available); Windows DPAPI**. Biometrics are a convenience
fast-path, not a passphrase replacement. Recovery is a BIP-39 phrase shown once at onboarding.

## Rationale

- Argon2id is the memory-hard PBKDF standard; chosen parameters resist offline cracking.
- Hardware-backed wrapping (Keystore/DPAPI) keeps the KEK off disk in usable form.
- "Lost passphrase + lost recovery phrase = permanently inaccessible vault" is intentional —
  there is no backdoor (blueprint §8.4).

## Consequences

- Passphrase required on first unlock per session, after repeated biometric failures, and after
  reboot (Android biometric key invalidation) — blueprint §8.3.
- Session auto-lock runs on a 60–300 s inactivity timer (PRD §4.6).
- This is high-stakes territory (CLAUDE.md V1 Phase 1) — changes require explicit review.
