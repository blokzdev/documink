# ADR-007: Multi-tenant schema from day one

- Status: Accepted
- Date: 2026-06-13
- Context: Recorded in V0 Phase 3; formalizes a decision already in the specs.

## Context

V1 is single-user, but the roadmap commits to Teams (V3+) with multiple members per workspace,
and to Projects as isolated scopes from V1. Retrofitting tenancy onto a single-user schema later
would be a costly, error-prone migration touching every table.

## Decision

Make the schema **multi-tenant-ready from day one**: every row carries a `workspace_id`
(V1: one user → one workspace; V3+ Teams: many members per workspace), and Project-scoped rows
also carry a `project_id` (NULL = global scope). **Project isolation is enforced at the
repository layer** — every service/repository method takes the scope and filters queries.
Cross-project access goes only through an explicit `AuditedCrossProjectAccess` API that writes to
`audit_log`. Primary keys are ULIDs (ordered, compact indexes).

## Rationale

- Forward compatibility for Teams with near-zero future migration cost (blueprint §3, §14.2).
- Project isolation is a hard cross-cutting invariant; enforcing it in repositories (not ad-hoc
  query sites) makes it testable (blueprint §3.3, §6.7; memory.md §6).

## Consequences

- No query may span projects without `AuditedCrossProjectAccess`; integration tests assert
  Project A data is invisible from a Project B scope.
- All six Mink memory tables follow the same `workspace_id`/`project_id` pattern (ADR-016).
