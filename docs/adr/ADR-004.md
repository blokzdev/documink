# ADR-004: GLiNER-PII-Edge as primary Tier 3 detection model

- Status: Accepted
- Date: 2026-06-13
- Context: Recorded in V0 Phase 3; formalizes a decision already in the specs.

## Context

The detection pipeline's Tier 3 is the always-on, bundled NER stage that catches PII that
regex (Tier 1) and ML Kit entity extraction (Tier 2) miss — names, organizations, locations,
and custom/zero-shot entity types. It ships inside the base APK (≤150 MB) and must run on-device
with a permissive license.

## Decision

Use **`knowledgator/gliner-pii-edge-v1.0` (Apache 2.0, ~100 MB INT8 ONNX)** as the primary
Tier 3 PII NER model, with the **OBI clinical de-id BERT (`obi/deid_bert_i2b2`, MIT, ~110 MB)**
for clinical PHI. Labels are zero-shot and live in `detection_labels.yaml` (not hard-coded),
so new entity types need no retraining. Loaded lazily via `flutter_onnxruntime` (ADR-002).

## Rationale

- GLiNER's zero-shot labeling supports custom entity types without retraining (blueprint §4.3).
- Apache 2.0 / MIT — compliant with the license policy; avoids the licensing traps
  (`iiiorg/piiranha-*` is CC-BY-NC-ND; anything trained on `ai4privacy/pii-masking-400k`
  inherits NC) — blueprint §2.3, §15 #1.
- INT8 fits the bundled-model budget; runs on the shared ONNX runtime.

## Consequences

- Inference is capped at 512 tokens; long documents are chunked with 50-token overlap and
  deduped (blueprint §4.3).
- ONNX models must be INT8-verified and SHA-256-pinned (blueprint §15 #12).
