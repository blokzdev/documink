# ADR-010: BYOC (Google Drive) + LAN mDNS as V1 sync

- Status: Accepted
- Date: 2026-06-13
- Context: Recorded in V0 Phase 3; formalizes a decision already in the specs.

## Context

V1 needs multi-device sync without DocuMink operating any backend (the first hosted component
arrives only in V4 — ADR-014). The transport must move encrypted CRDT deltas (ADR-009) while the
user — not DocuMink — owns the infrastructure, and must also work offline on a local network.

## Decision

Ship two V1 sync transports:

1. **BYOC to Google Drive** via the `drive.appdata` scope (app-private folder, invisible in the
   user's Drive UI). Stores `vault.db.enc` (SQLCipher) + `deltas/<ulid>.cbor.enc` (AES-GCM CRDT
   deltas). OAuth via `google_sign_in`; tokens in platform secure storage.
2. **LAN sync via mDNS** (`_documink._tcp.local.`): devices advertise `device_id` + public-key
   fingerprint; first pairing exchanges a transport key by QR; transport is WebSocket-over-TLS
   (self-signed, pinned from the QR), AES-GCM-framed.

## Rationale

- BYOC means the user supplies storage (their Drive quota) — no DocuMink egress or data at rest,
  consistent with the no-backend-until-V4 stance (PRD §4.8).
- LAN mDNS gives fully offline device-to-device sync with no cloud dependency (blueprint §9.3).
- Never sync plaintext — not even in dev builds (blueprint §9.1, §15 #5).

## Consequences

- V3+ adds more BYOC providers (iCloud post-V4, OneDrive, Dropbox) behind a `CloudProvider`
  interface; WAN P2P arrives via the V4 relay (ADR-014) — blueprint §9.5.
- Sync keys are always encrypted; transport key derives from the MK (ADR-005).
