# ADR-020: Key-storage layout — salt & biometric KEK in secure storage, dedicated DB-key subkey

- Status: Accepted (**Decision 1 superseded by [ADR-023](ADR-023.md)** — the salt now lives in a
  plaintext file, not `flutter_secure_storage`, after a real-device Keystore-key-loss bricked the vault)
- Date: 2026-06-13
- Context: V1 Phase 1b (KeyService). Refines ADR-005; corrects blueprint §8.1 / §8.2.

## Context

ADR-005 and blueprint §8.1 state the Argon2id **salt lives in `vault_meta`**, and §8.2 opens the
SQLCipher database with "the derived key". Implementing KeyService surfaced a bootstrap paradox:

- `vault_meta` is a table **inside** the SQLCipher-encrypted database (schema landed in 1a, §3.1).
- Opening that database requires the DB key, which is derived from the Master Key, which is derived
  from the passphrase **and the salt**.
- Therefore the salt cannot be read from `vault_meta` before unlock — it would already need the key
  it helps derive. The salt must live somewhere readable *pre-unlock*.

The spec was also silent on *which* derived key opens SQLCipher: §8.1 lists three subkeys (KEK,
fingerprint-HMAC, sync), none of them a database key.

## Decision

1. **Pre-unlock material → `flutter_secure_storage`** (Android Keystore / Windows DPAPI-backed),
   which is readable before the vault opens. In V1 P1b this is the **Argon2id salt** (base64).
   The salt is not secret, but it must survive launches and be available pre-unlock. The
   **biometric-gated wrapped KEK** (ADR-005) will also live here when Phase 5 adds biometrics.
2. **`vault_meta` holds only post-unlock material** — the wrapped DEK (AES-256-GCM under the KEK)
   and `key_version`. It is read after the DB is open.
3. **A dedicated DB-key subkey** opens SQLCipher: HKDF-SHA256 with info `documink:sqlcipher:v1`,
   distinct from the KEK (`documink:kek:v1`). Database encryption and DEK-wrapping never share key
   material. The full subkey set is now four: DB key, KEK, fingerprint-HMAC key, sync key.
4. **Passphrase-only in V1 P1.** The KEK is re-derived from the MK on every unlock, so nothing
   key-secret is persisted in P1b; only the (non-secret) salt is. The Keystore-wrapped KEK is the
   biometric fast-path and is deferred to Phase 5 (blueprint §8.3).

## Rationale

- Resolves the paradox without weakening anything: a salt is public input to a KDF; storing it in
  the platform secure store is strictly stronger than a plaintext sidecar and keeps all vault
  metadata that *is* sensitive (wrapped DEK) inside the encrypted DB.
- Domain-separated subkeys (RFC 5869 `info`) mean a compromise/rotation of one role never exposes
  another; giving SQLCipher its own key avoids overloading the KEK.
- Keeps KeyService **database-free and pure-Dart testable** — it derives keys and wraps/unwraps the
  DEK as pure functions; `VaultService` (1c) owns the DB handle and persists the wrapped DEK.

## Consequences

- `lib/services/key_service.dart` derives MK (Argon2id, params pinned to §8.1, version 0x13),
  four HKDF-SHA256 subkeys, and generates/wraps/unwraps the DEK (AES-256-GCM). The salt is read
  from / created in a `SecureKeyStore` (`lib/services/secure_key_store.dart`), faked in tests.
- Correctness is anchored to RFC 9106 (Argon2id) and RFC 5869 (HKDF-SHA256) known-answer tests;
  DocuMink wiring (domain strings, DEK wrapping) is pinned with regression vectors.
- **MK/subkey zeroization is best-effort** on managed Dart (the GC may retain unscrubable copies);
  keys are held in `Uint8List`, overwritten on lock, and never placed in `String`s or logs.
- If V1.x device-profiling (Phase 9) tunes Argon2 cost, the chosen parameters must be persisted
  alongside the salt so derivations stay reproducible — out of scope here (cost is fixed in V1).

## Alternatives considered / rejected

- **Plaintext sidecar file for the salt** — acceptable (a salt isn't secret) but strictly weaker
  than the platform secure store and adds a second on-disk artifact to manage. Rejected.
  **(Reversed by [ADR-023](ADR-023.md): "strictly weaker" was wrong — the platform secure store can
  silently lose its Keystore key and brick the vault, which a plaintext salt cannot. The sidecar is
  now the chosen design.)**
- **Keep salt literally in `vault_meta`** — impossible without abandoning Argon2id-as-KDF and using
  SQLCipher's own header-salt/passphrase KDF, which contradicts ADR-019's raw-key design. Rejected.
- **Reuse the KEK as the SQLCipher key** — fewer subkeys, but conflates DB encryption with DEK
  wrapping. Rejected for clean separation.
