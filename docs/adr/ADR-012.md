# ADR-012: Mink as a single-agent-per-user scoped-context model (not multi-agent)

- Status: Accepted
- Date: 2026-06-13
- Context: Recorded in V0 Phase 3; formalizes a decision already in the specs.

## Context

Mink's memory design adapts the MIRIX taxonomy (Wang & Chen, 2025), which proposes a
**multi-agent** architecture — eight LLMs (six memory managers + meta + chat). On tiered
on-device hardware, eight cold-start LLM invocations per response would destroy latency and
battery life on anything below flagship tier.

## Decision

Mink is a **single LLM agent per user** with **scoped context**, backed by whichever Tier 4 model
the active device can run (ADR-011); identity is constant across paired devices while the
inference model adapts per device. Memory operations route through a **deterministic code router
(~200 lines of Dart, no LLM)** that dispatches named tool calls to per-type repositories. This
keeps MIRIX's taxonomic benefits without multi-agent overhead.

## Rationale

- On-device tractability: one model invocation per turn, not eight (memory.md §4).
- Determinism: routing in code is fast, debuggable, and testable — not subject to LLM variance
  (memory.md §1.2, §4.1).

## Alternatives considered / rejected

- **MIRIX multi-agent (one agent per memory type)** — rejected for cold-start latency/battery on
  mobile; if a PR proposes separate agents per memory type, reject with reference to this ADR
  (blueprint §15 #27).

## Consequences

- Binding architecture invariant: no multi-agent Mink (`architecture-invariants.md`).
- Memory taxonomy and the router are specified authoritatively in `memory.md` (ADR-016).
- High-stakes territory (CLAUDE.md V1 Phase 12 — Mink memory).
