# ADR-022: Tier 3 detection model — hybrid bundled-baseline + device-tiered downloaded upgrade

- Status: Accepted
- Date: 2026-06-13
- Context: V1 Phase 2 (detection). Extends ADR-004; relates to ADR-011/ADR-017 (tiered models + signed manifest). Maintainer-confirmed.

## Context

ADR-004 / blueprint §4.3 specify Tier 3 as a **single bundled** GLiNER-PII INT8 ONNX model in the
base APK. Researching the actual `knowledgator/gliner-pii-*` family (all Apache-2.0, 60+ PII
categories, zero-shot — fits our updateable `detection_labels.yaml`) surfaced two problems with
"bundle one model":

- **Size vs the install budget.** Variants ladder by size/accuracy: edge (~F1 75.5%), small
  (~76.8%), base (UINT8 ~197 MB / FP16 ~330 MB, F1 81.0%), large (F1 83.3%). PRD exit criteria cap
  **install-with-bundled-models at ≤250 MB**, and Phase 12 also bundles a ~80 MB MiniLM embedding
  model. Bundling base/large is infeasible; bundling only a small variant caps everyone at ~75–77%.
- **One model can't be optimal for all devices.** Low-end devices need the small/edge model;
  capable devices can run base/large (ideally FP16, since 8-bit GLiNER quantization has a
  documented accuracy drop) for materially higher F1.

DocuMink already has the machinery to solve this — the Device Capability Profiler + Ed25519-signed
manifest + SHA-256 + Play-Asset-Delivery/download with graceful degradation (ADR-011/017, §4.7,
roadmap Phase 9) — built for Tier 4 LLMs.

## Decision

**Hybrid, device-capability-tiered Tier 3 detection models:**

1. **Bundle the smallest viable GLiNER-PII variant** (edge/small, ~100 MB INT8) in the base APK as
   an always-available **offline baseline**, keeping base APK ≤150 MB and install ≤250 MB. Exact
   variant + quantization finalized at implementation against the measured budget.
2. **Offer a downloaded upgrade** (base/large, FP16 where the device allows) on capable devices,
   selected by the **Device Capability Profiler** and delivered through the **same signed manifest +
   SHA-256 + PAD/download** path as Tier 4 (extends the manifest to cover Tier 3 detection models).
3. **Graceful degradation** (already the pipeline's behavior): if the upgrade isn't downloaded or
   the device is below threshold, Tier 3 runs the bundled baseline; if even that is unavailable,
   detection proceeds with **Tiers 1–2 only**. No hard failure.

## Consequences

- **Sequencing:** the downloaded-upgrade path depends on the Phase 9 profiler/manifest/download
  infrastructure, so full Tier 3 lands **with/after Phase 9**. Phase 3 (Anonymizer operators) is
  done next; the bundled-baseline recognizer is implemented when Tier 3 is wired (Phase 9-adjacent).
- blueprint §4.3 updated (hybrid, not single-bundle); roadmap Phase 2 Tier 3 task + exit criteria
  annotated; `models.md` notes the manifest now also serves Tier 3 detection models (catalog
  finalized at Phase 9). ADR-004 stands; this refines its delivery model.
- `verify-model-hashes` will register the bundled baseline's SHA-256 when it lands; downloaded
  variants are verified against the signed manifest at fetch time (no fabricated hashes now).
- Re-evaluate whether the clinical PHI model (`obi/deid_bert_i2b2`, ~110 MB, ADR-004) should also be
  download-on-demand rather than bundled — bundling both NER models pressures the 250 MB ceiling.

## Alternatives considered / rejected

- **Bundle one small INT8 model (original §4.3).** Simplest/offline, but caps accuracy at ~75–77%
  for all devices and still pressures the budget. Rejected as the *whole* strategy; kept as the
  bundled baseline within the hybrid.
- **Download-only (nothing bundled).** Smallest APK, but no offline Tier 3 until first download —
  worse first-run/airplane-mode UX for a privacy-first local app. Rejected.
