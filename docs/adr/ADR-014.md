# ADR-014: V4 thin relay backend as DocuMink's first and only hosted component, scoped to encrypted transport for three uses (URL sharing, community templates, same-user WAN inference)

- Status: Accepted
- Date: 2026-06-13
- Context: Recorded in V0 Phase 3; formalizes a decision already in the specs.

## Context

DocuMink is on-device and backend-free through V3. V4 introduces features that inherently need a
network rendezvous point — URL-based document sharing, community template distribution, and
same-user inference dispatch over the internet (WAN). The risk is that "add a backend" quietly
becomes "move data and inference to the cloud," violating the core privacy stance.

## Decision

Introduce a **thin relay backend in V4 — DocuMink's first and only hosted component** — minimal,
stateless, content-addressed, with no user accounts and no persistence beyond expiry. It serves
exactly three uses, all **end-to-end encrypted (server sees ciphertext only**, except public
author-signed template bytes):

1. **URL-based encrypted document sharing** — key lives in the URL fragment, never sent to the server.
2. **Community template publishing** — author-signed blobs; DocuMink hosts bytes, makes no endorsement.
3. **Same-user WAN inference dispatch** — relays encrypted inference between the user's own devices (opt-in Pro).

**The relay never performs inference itself.** All inference remains on-device, always
(blueprint §15 #23).

## Rationale

- Some V4 features need a rendezvous; scoping the relay to encrypted transport preserves the
  no-cloud-inference invariant (blueprint §9.6, §14.3).
- Stateless + content-addressed + no accounts minimizes the attack surface and data-at-rest.

## Consequences

- Hosting is object storage + edge functions (e.g. Cloudflare Workers + R2); anonymous uploads
  with rate limiting and expiry.
- This is the only DocuMink-operated infrastructure besides the signed `manifest.json` (ADR-017).
