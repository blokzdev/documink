# ADR-003: SQLCipher + drift for vault

- Status: Accepted
- Date: 2026-06-13
- Context: Recorded in V0 Phase 3; formalizes a decision already in the specs.

## Context

DocuMink is local-first: the database lives on the user's device, not infrastructure we control.
It must hold the reversible-tokenization vault and everything else (documents, entities, Mink
memory, audit log) encrypted at rest, with full-text and vector search, JSON, and CRDT sync —
all within a 250 MB install budget that also carries models, down to 2 GB-RAM devices.

## Decision

Use **SQLite encrypted with SQLCipher (AES-256)** as the single on-device database, accessed via
**drift (MIT)** with compile-time-typed reactive queries and versioned migrations.
`sqlcipher_flutter_libs` (MIT) provides the Android + Windows binaries. One encrypted database,
one backup artifact, one sync target. Capability extensions: FTS5 (search), sqlite-vec (vectors),
JSON1 (manifests/payloads), cr-sqlite (sync — ADR-009).

## Rationale

- SQLite is the category-correct embedded engine; its extensions cover every capability we need
  (FTS5 ≈ Elasticsearch BM25; sqlite-vec ≈ pgvector/Pinecone; JSON1 ≈ JSONB).
- drift gives type-safe queries and migrations; chosen over `sqflite_sqlcipher`, which lacks
  Windows binaries (blueprint §15 #10).

## Alternatives considered / rejected

- **Postgres / embedded-Postgres** — a server process, 50–100+ MB binary, 100+ MB idle RAM;
  incompatible with sandboxed mobile lifecycles (blueprint §2.4, §15 #25).
- **Specialized DBs** (Elasticsearch, Pinecone, Redis, MongoDB, InfluxDB) — redundant with
  SQLite extensions; rejected (§15 #26).
- **Hive** — values encrypted but keys stored plaintext (NVISO Labs 2024); forbidden (§15 #2).

## Consequences

- Everything persists in one SQLCipher file; no separate vector/time-series/document DB.
- Key management for the SQLCipher key is defined separately (ADR-005).
- `*.g.dart` from drift codegen is committed and CI-checked for freshness; codegen needs `--force-jit`.
