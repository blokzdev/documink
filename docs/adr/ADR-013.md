# ADR-013: Projects as declarative domain-agnostic harnesses delivered via signed remote manifest

- Status: Accepted
- Date: 2026-06-13
- Context: Recorded in V0 Phase 3; formalizes a decision already in the specs.

## Context

Users work across domains (medical, legal, tax, creative, engineering, …) with different
detection policies, tool permissions, custom entity types, and Mink personas. Hard-coding domain
behaviors into the app would make every new domain an app release and bloat the binary.

## Decision

Model a Project as a **declarative, domain-agnostic JSON manifest** (`projects.manifest_json`,
versioned) holding permissions, default policy, custom entity types, Mink persona/prompt
addendum, expected file types, and UI accents. Eight **Verified templates** (personal, medical,
legal, tax, research, creative, engineering, blank) ship in an **Ed25519-signed remote manifest**
at `documink.ai/templates/manifest.json`, fetched at first launch and refreshed weekly with a
bundled offline fallback. Three creation paths: from template, AI-scaffolded from uploaded docs,
or a guided blank wizard.

## Rationale

- Declarative manifests make domains data, not code — new/updated templates ship without an app
  release (blueprint §6.1, §6.4).
- Signing reuses the same Ed25519 infrastructure as the model manifest; signature-verification
  failure blocks updates and never falls back to unsigned (blueprint §6.4).

## Consequences

- Project isolation is enforced per ADR-007 (repository-layer `workspace_id`/`project_id`).
- AI-scaffolded templates must be labeled and re-reviewable — never treated as Verified
  (blueprint §15 #22).
- V4 community templates ride the thin relay (ADR-014), signed by their author, not by DocuMink.
