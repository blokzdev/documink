# ADR-002: ONNX Runtime via `flutter_onnxruntime` as primary inference runtime for encoders

- Status: Accepted
- Date: 2026-06-13
- Context: Recorded in V0 Phase 3; formalizes a decision already in the specs.

## Context

Tiers 1–3 of the detection pipeline and Mink's memory vector search rely on encoder-style
models: GLiNER-PII-Edge (NER), the OBI clinical de-id BERT, and the always-bundled
`all-MiniLM-L6-v2` embedding model. These must run on-device across Android and Windows with a
single model artifact and hardware acceleration where available. (Generative Tier 4 LLMs use a
separate runtime path — see ADR-011.)

## Decision

Use **ONNX Runtime via the `flutter_onnxruntime` plugin (MIT)** as the primary inference runtime
for encoder/embedding models. Ship one ONNX file per model for both platforms; let the runtime
pick the execution provider: NNAPI → XNNPACK → CPU on Android, DirectML → CPU on Windows.

## Rationale

- One shared INT8 ONNX artifact for both platforms; no per-platform model files (blueprint §10.3).
- `flutter_onnxruntime` 1.5.1+ is 16 KB page-size compliant, required for Pixel 9+/Android 15+.
- MIT licensed; broad EP support gives NPU/GPU acceleration without app-side branching.

## Alternatives considered / rejected

- MediaPipe LLM Inference API — deprecated on Android/iOS in 2025 (blueprint §15 #9).
- Per-platform native runtimes — rejected to keep one model artifact and one abstraction.

## Consequences

- Encoder models must be INT8-quantized and SHA-256 verified (blueprint §15 #12).
- Generative LLMs use LiteRT-LM (`flutter_gemma`) / llama.cpp (`fllama`) instead (ADR-011).
