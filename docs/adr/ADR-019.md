# ADR-019: Encrypted vault via `package:sqlite3` v3 user-define (`source: sqlite3mc`)

- Status: Accepted
- Date: 2026-06-13
- Context: V1 Phase 1a (core data layer). Amends ADR-003.

## Context

ADR-003 / blueprint §2.4 specify an AES-256 SQLCipher-encrypted SQLite vault and name
`sqlcipher_flutter_libs` (MIT) as the integration package providing the encrypted native binaries
and the classic `open.overrideFor(OperatingSystem.android, openCipherOnAndroid)` runtime recipe.

In the repo's actually-resolved dependency stack (drift 2.32.1 → `package:sqlite3` 3.3.3),
`sqlcipher_flutter_libs` resolves to `0.7.0+eol`, which is an **empty no-op shim**: its own
documentation states "This package does not do anything … Starting from version 3.x [of
`package:sqlite3`], this separate package is no longer necessary." The classic recipe therefore no
longer exists in this stack. `package:sqlite3` 3.x instead selects an encrypted SQLite build at
compile time through Dart/Flutter **Build Hooks user-defines**.

## Decision

Provide the encrypted vault by adding to `pubspec.yaml`:

```yaml
hooks:
  user_defines:
    sqlite3:
      source: sqlite3mc   # SQLite3 Multiple Ciphers (MIT)
```

`package:sqlite3` then bundles the SQLite3 Multiple Ciphers build automatically on every isolate;
no runtime `open` override is required. The vault is opened with the raw 32-byte key derived by
KeyService passed as a hex blob (`PRAGMA key = "x'…'"`), with the **SQLCipher-compatible cipher
pinned explicitly** (`PRAGMA cipher = 'sqlcipher'`) to honor the AES-256 SQLCipher intent of
ADR-003. `sqlcipher_flutter_libs` is removed from the dependency set.

## Rationale

- It is the **maintained** path for encryption on `package:sqlite3` 3.x; the named package is dead.
- **No dependency downgrade or version pin** — consistent with CLAUDE.md's hard rule against
  downgrading/pinning to dodge Build Hooks. The Build-Hooks machinery is already in our tree
  (drift → sqlite3 → native_toolchain_c) and codegen already runs under `--force-jit`.
- **License-clean:** SQLite3 Multiple Ciphers is MIT and (unlike the SQLCipher community build)
  does not link OpenSSL. The native build is bundled via the hook, not a pub.dev package, so it is
  not visible to the license-scan job, but it is allow-list-clean under `license-policy.md`.

## Alternatives considered / rejected

- **`source: sqlcipher`** (SQLCipher community build) — closest to the literal package name and
  AES-256-CBC, but links OpenSSL on Android/Linux/Windows (heavier binary) and may track an older
  SQLite version. Rejected in favor of the lighter, OpenSSL-free sqlite3mc.
- **Downgrade to `package:sqlite3` 2.x + `sqlcipher_flutter_libs` 0.6.0 + drift <2.32** — keeps the
  classic recipe but pins/downgrades the core DB stack to dodge Build Hooks, violating the
  CLAUDE.md hard rule, and locks us to EOL tooling. Rejected.

## Consequences

- `lib/data/sqlcipher_executor.dart` opens `NativeDatabase.createInBackground` and runs
  `PRAGMA cipher` + `PRAGMA key` in `setup:`, guarding that an encrypted build is linked
  (`PRAGMA cipher_version` non-empty).
- Keyed-DB integration tests need the encrypted host build (fetched/compiled by the hook); schema
  tests use an in-memory executor. blueprint §2.4 updated to reflect the mechanism.
- If `sqlcipher_flutter_libs` is ever revived for the v3 toolchain, re-evaluate at a phase boundary.
