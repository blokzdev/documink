# ADR-018: Defer the `mink_embeddings` vec0 virtual table to V1.2

- Status: Accepted
- Date: 2026-06-13
- Context: V1 Phase 1a (core data layer). Amends ADR-003.

## Context

blueprint §3.2 defines a `mink_embeddings` virtual table (`CREATE VIRTUAL TABLE … USING vec0`)
backed by the sqlite-vec extension, storing `FLOAT[384]` vectors for Semantic and Resource memory
similarity search. Phase 1 builds the encrypted data layer and full schema. The original phase
plan intended to create this table at schema version 1.

Verification during planning found that sqlite-vec cannot be cleanly bundled into a Flutter app
under the encrypted SQLite library today: there is no published Dart/Flutter package shipping
sqlite-vec native libraries; the only binding is unmerged, ships no arm64-Android prebuilts, and
its author declared the approach obsolete pending Dart native-assets/Build-Hooks support — the
same toolchain area this repo already works around (dart-lang/build#4343). Statically linking
`vec0` into the encrypted build is a per-ABI native-build project of its own (NDK, 16 KB
page-size compliance), out of scope for a data-layer phase.

## Decision

The V1 Phase 1 schema creates **all 16 relational tables** (§3.1 + §3.2) but **does NOT** create
the `mink_embeddings` vec0 virtual table. `mink_semantic_memory.embedding_id` is implemented as a
**plain nullable `TEXT` column with no foreign key** (a vec0 virtual table cannot be a FK target),
keeping the column forward-compatible. The vec0 table is created when V1.2 activates embedding
search, alongside its consumers.

## Rationale

- blueprint §3.2 already marks the only consumers — Semantic Memory and Resource Memory — as
  "schema V1, **activation V1.2**." Nothing in V1 Phase 1 (workspaces/projects/documents/entities/
  tokens/vault) reads or writes embeddings, so deferring the table changes no V1 behavior.
- Honors the privacy and architecture invariants without taking on a speculative native-build
  dependency in the highest-stakes phase.

## Alternatives considered / rejected

- **Ship vec0 now via a bundled sqlite-vec native build** — a separate per-ABI native project;
  disproportionate risk/scope for a phase that has no embedding consumers yet.
- **Drop `embedding_id` entirely** — would force a schema migration in V1.2; keeping it as a plain
  TEXT column avoids that.

## Consequences

- A schema test asserts `mink_embeddings` is **absent** and that `embedding_id` accepts an
  arbitrary value (no FK). V1.2 must add the vec0 table and, if desired, a backfill.
- The standing workaround note (sqlite-vec activation) should be re-evaluated at the V1.2
  phase boundary when Dart native-assets/Build-Hooks maturity may make bundling viable.
