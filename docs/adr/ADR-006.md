# ADR-006: FF1 (not FF3) for format-preserving encryption

- Status: Accepted
- Date: 2026-06-13
- Context: Recorded in V0 Phase 3; formalizes a decision already in the specs.

## Context

Reversible anonymization needs a format-preserving mode for fields where downstream systems
expect a specific shape — credit-card numbers, EINs, phone numbers. Format-preserving encryption
(FPE) keeps the format while encrypting the value, so it can be reversed via the vault. NIST
defines two schemes, FF1 and FF3/FF3-1.

## Decision

Implement **NIST SP 800-38G FF1 only**, as ~300 lines of Dart over `pointycastle` AES, validated
against NIST test vectors. The FPE tweak is `entity_type + workspace_id` hash. **Do NOT implement
FF3 / FF3-1.** For credit cards, preserve the 6-4-4-4 format and keep last-4 (or first-6) in
clear per policy.

## Rationale

- NIST **withdrew FF3** in Draft Revision 2 (February 2025) over cryptographic weaknesses; FF1 is
  the surviving standard (blueprint §7.1, §15 #3).
- A small hand-rolled FF1 on a vetted AES primitive avoids pulling a heavier crypto dependency
  and is unit-testable against published NIST vectors.

## Alternatives considered / rejected

- **FF3 / FF3-1** — withdrawn by NIST; rejected outright.

## Consequences

- FPE is one of three reversible modes (with Random Token and AES-GCM encrypt-in-place); all
  reversible modes round-trip through the biometric-gated decode flow (ADR-005, blueprint §7.2).
- FPE is in scope for the pre-launch security audit and FPE fuzzing (blueprint §12.4).
