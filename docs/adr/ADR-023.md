# ADR-023: Argon2id salt moves to a plaintext file, off the platform Keystore

- Status: Accepted
- Date: 2026-06-19
- Context: V1 Phase 1 hardening. **Supersedes ADR-020 Decision 1** for the salt; reverses ADR-020's
  "plaintext sidecar rejected" alternative. Corrects blueprint §8.1.

## Context

ADR-020 stored the Argon2id **salt** in `flutter_secure_storage` (FSS), reasoning that the platform
secure store is "strictly stronger than a plaintext sidecar". A real-device bug report disproved the
"stronger, and free" assumption: on a Samsung device, FSS's Android-Keystore RSA key
(`…FlutterSecureStoragePluginKeyOAEP`) was **lost between launches** —

```
keystore2: ResponseCode(7) (KEY_NOT_FOUND) … FlutterSecureStoragePluginKeyOAEP
FlutterSecureStorage: InvalidKeyException: Failed to unwrap key
```

— with the device's StrongBox Keymaster logged `non-responsive`. The encrypted salt blob survived in
SharedPreferences but became **undecryptable**, so unlock derived the wrong DB key and SQLCipher
refused to open: the vault was **permanently bricked even with the correct passphrase**, and onboarding's
"create" had already failed for the same reason.

Crucially, the salt is **not secret** (blueprint §8.1; ADR-020 itself says so): it is public input to a
KDF whose only job is to be unique and survive launches. Putting a non-secret value behind a hardware
key that can vanish converts a transient Keystore failure into unrecoverable data loss — a single point
of failure with no security upside.

## Decision

1. **The Argon2id salt lives in a plaintext app-private file** (`vault.salt`, sibling of `vault.db` in
   the application-support directory), via a `SaltStore` seam (`FileSaltStore`). It is read **before**
   unlock and never depends on the Keystore.
2. **FSS leaves the V1 critical path.** It is retained only as a one-way **migration read-fallback**:
   `KeyService.readSalt()` reads a legacy FSS salt (if the Keystore still works) and writes it to the
   file, so installs created before this change keep working. All legacy reads are best-effort and
   tolerate a broken Keystore (treated as "no salt"). FSS remains wired for the **Phase-5 biometric
   wrapped KEK** (ADR-005), which genuinely needs hardware binding.
3. **Atomic create + recovery.** A failed `initialize()` now rolls back **both** artifacts (salt file +
   `vault.db`); `vaultExists()` requires **both** present and self-heals a half-created state; and a
   destructive **`reset()`** ("reset & start over") lets a user erase an unopenable vault and restart.

## Rationale

- A salt is public by design (RFC 9106): storing it in plaintext leaks nothing an attacker with the
  device doesn't already have. The SQLCipher database remains the encrypted artifact; **its key is
  never stored** — only re-derived from passphrase + salt. So security is unchanged, and the
  brick-the-vault failure mode is eliminated.
- Removing the Keystore from the create/unlock path also fixes the **"couldn't create vault"** failure,
  which was the same Keystore-write flakiness manifesting at enrollment.
- Keeps `KeyService` pure-Dart testable (the file seam is faked/temp-filed in tests).

## Consequences

- `lib/services/salt_store.dart` (`SaltStore` + atomic `FileSaltStore`); `KeyService` takes a
  `SaltStore` + optional legacy `SecureKeyStore`; bootstrap wires `vault.salt`.
- `VaultService` gains `reset()` + `_wipeVaultArtifacts()` and a combined, self-healing `vaultExists()`.
- The salt is no longer hardware-encrypted at rest. Accepted: it is non-secret, and the threat model
  never relied on its confidentiality.
- Device verification (`VERIFICATION.md`): create→relaunch→unlock on the affected device; reset flow;
  migration from an install whose FSS salt is still readable.

## Alternatives considered / rejected

- **Keep FSS, add a file mirror/fallback** — survives Keystore loss on *read*, but the create path
  still attempts a Keystore *write* that can fail on flaky devices (the observed "couldn't create
  vault"). More moving parts, no upside. Rejected.
- **Only harden `AndroidOptions`** (`encryptedSharedPreferences`/`resetOnError`) — still funnels through
  a Keystore master key that can be invalidated; `resetOnError` would silently wipe the salt (= brick).
  Does not remove the SPOF. Rejected.
