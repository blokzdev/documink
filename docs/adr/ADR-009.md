# ADR-009: cr-sqlite for CRDT sync

- Status: Accepted
- Date: 2026-06-13
- Context: Recorded in V0 Phase 3; formalizes a decision already in the specs.

## Context

Mink and the vault sync across a user's paired devices with no DocuMink-controlled server in the
loop (ADR-010, ADR-014). Multi-device edits must merge without a central coordinator and without
ever syncing plaintext. The merge engine must run inside the embedded SQLite database (ADR-003).

## Decision

Use **cr-sqlite (MIT)** — a SQLite extension that turns tables into CRDTs — as the conflict
resolution engine. Sync ships **idempotent, AES-GCM-encrypted CRDT deltas**. Scalars resolve
last-write-wins; collections by set union; memory types carry per-type semantics (e.g. episodic
= append-only set union; semantic entities merge by fingerprint with monotonic counts).

## Rationale

- cr-sqlite is mature, mobile-ready, and runs in-process in our existing SQLite file — no
  separate sync service. There is no embedded-tier Postgres equivalent (blueprint §2.4).
- CRDTs give conflict-free multi-device merges without a server (blueprint §9.1).

## Consequences

- Hard conflicts that CRDTs can't auto-resolve (e.g. same custom-entity label, different pattern)
  surface in Settings → Sync Conflicts (blueprint §9.4).
- All six memory types sync over this channel with type-specific conflict rules (memory.md §9).
